# Push_swap プロジェクト要件定義書

## 1. プロジェクト概要

### 1.1 プロジェクト名
Push_swap - スタックソーティングアルゴリズム実装プロジェクト

### 1.2 目的
2つのスタックと限定された操作を使用して、整数配列を昇順にソートするプログラムを開発する。

### 1.3 開発体制
- **プロジェクトマネージャー（PM）**: 全体統括・進捗管理
- **アーキテクト（ARCH）**: システム設計・技術選定
- **アルゴリズムエンジニア（ALGO）**: ソートアルゴリズム設計・最適化
- **コアデベロッパー（CORE）**: 基本機能実装
- **テストエンジニア（TEST）**: テスト設計・実装
- **品質保証エンジニア（QA）**: Norm準拠・品質チェック

### 1.4 現在の実装状況
- 基本的なスタック操作は実装済み
- 入力検証とエラーハンドリング実装済み
- libftライブラリ統合済み
- ソートアルゴリズムの最適化が必要

## 2. 機能要件

### 2.1 基本機能
```
FR-001: 整数配列を引数として受け取り、ソート操作を出力する
FR-002: 11種類のスタック操作を実装する（sa, sb, ss, pa, pb, ra, rb, rr, rra, rrb, rrr）
FR-003: 最小操作数でソートを完了する
FR-004: エラー時は標準エラーに"Error\n"を出力する
FR-005: 引数なしの場合は何も出力せずに終了する
```

### 2.1.1 エラー処理詳細
- 重複値の検出
- 整数オーバーフロー/アンダーフロー（INT_MIN～INT_MAX）
- 無効な文字列（"42a", "++42", "--42"等）
- 空文字列、空引数（""）
- メモリ割り当て失敗時の適切な処理

### 2.2 ボーナス機能
```
FR-B001: checkerプログラムで操作列の妥当性を検証する
FR-B002: 標準入力から操作を読み込み実行する
FR-B003: ソート完了時は"OK"、それ以外は"KO"を出力する
```

### 2.3 技術仕様
- **スタック実装**: 動的配列方式（拡張性とパフォーマンスのバランス）
- **整数範囲**: INT_MIN（-2147483648）～ INT_MAX（2147483647）
- **入力形式**: 
  - 個別引数: `./push_swap 3 2 1`
  - 文字列引数: `./push_swap "3 2 1"`
- **メモリ管理**: libftのft_calloc/freeを使用
- **出力形式**: 各操作を改行区切りで標準出力

## 3. 非機能要件

### 3.1 性能要件
```
NFR-001: 3要素を最大3操作でソート
NFR-002: 5要素を最大12操作でソート
NFR-003: 100要素を700操作以内でソート（100%達成）
NFR-004: 500要素を5500操作以内でソート（100%達成）
NFR-005: メモリリークがないこと
```

### 3.1.1 最小達成基準（80%グレード）
- 100要素: 1300操作以内 かつ 500要素: 5500操作以内
- または、100要素: 700操作以内 かつ 500要素: 11500操作以内
- または、100要素: 1100操作以内 かつ 500要素: 8500操作以内

### 3.2 品質要件
```
NFR-006: 42 Norm Version 4に完全準拠
NFR-007: セグメンテーションフォルトなどの異常終了がないこと
NFR-008: Makefileが適切に動作すること（不要な再リンクなし）
NFR-009: 時間計算量 O(n log n)、空間計算量 O(n)を目標
```

## 4. 制約事項
- C言語で実装
- グローバル変数は禁止
- 使用可能な外部関数: read, write, malloc, free, exit, ft_printf（またはカスタム実装）
- 42 Norm準拠（関数25行以内、1ファイル5関数まで、1行80文字以内）

1. 要件定義・基本設計: 2日
2. 詳細設計: 2日
3. 実装: 5日
4. テスト: 3日
5. 最適化・品質保証: 3日

### 5.1 マイルストーン
- Day 4: インターフェース凍結
- Day 9: アルゴリズム実装完了
- Day 12: 統合テスト開始
- Day 15: 最終成果物提出

## 6. テスト戦略

### 6.1 単体テスト
- 各スタック操作（sa, sb等）の個別テスト
- 境界値テスト（INT_MIN, INT_MAX, 0）
- エラーケース網羅テスト

### 6.2 統合テスト
- checker_linuxとの連携テスト
- 既存のtest.shスクリプトの活用
- ランダムデータによる網羅的テスト（Python生成）

### 6.3 パフォーマンステスト
- 操作数カウントの自動化
- 統計的な性能評価（平均、最悪ケース、最良ケース）
- メモリ使用量の計測（valgrind）

## 7. リスク管理

| リスク | 影響度 | 対策 |
|--------|--------|------|
| アルゴリズム性能未達成 | 高 | 複数アルゴリズムの並行開発・比較 |
| Norm違反 | 中 | 開発初期からのnorminette定期実行 |
| メモリリーク | 高 | valgrindの日次実行、適切なfree |
| 統合の遅延 | 中 | 早期インターフェース確定、モックテスト |

### 8.1 ブランチ戦略
- main: 安定版（評価提出用）

### 8.2 コードレビュープロセス

- CI/CDパイプライン（norminette, valgrind, 基本テスト）

### 8.3 日次同期
- スタンドアップミーティング（進捗共有）
- ブロッカーの即時共有
- 成果物の段階的統合

## 9. モジュール構成と依存関係

```
push_swap/
├── src/
│   ├── main.c              [CORE] エントリーポイント
│   ├── operations/         [CORE] スタック操作
│   ├── sort/               [ALGO] ソートアルゴリズム
│   ├── stack_basic/        [ARCH] 基本スタック機能
│   ├── utils/              [ARCH] ユーティリティ
│   └── utils_libft/        [既存] libftライブラリ
├── includes/               [ARCH] ヘッダーファイル
└── tests/                  [TEST] テストコード
```

### 9.1 依存関係
- libft → 全モジュール
- stack_basic → operations → sort
- utils → main, sort
- エラーハンドリングは全モジュールで統一インターフェース使用

## 10. ドキュメント標準

### 10.1 コード規約
- 関数コメント: 目的、引数、戻り値を明記
- 複雑なロジックには処理説明を追加
- 変数名は意味が明確なものを使用

### 10.2 Git規約
- コミットメッセージ: `[役割] 動詞: 内容`（例: `[ALGO] Add: radix sort implementation`）
- PR説明: 変更内容、テスト結果、レビューポイントを明記
